# This file contains the definitions used to create the options used by the code
# formatter (and autocomplete, and a few other things). The file is fed into
# formatter_options.pl, which generates the options by processing inlined Perl
# in the template files. Here are the templates currently in use:
#
# DefaultCodeFormatterConstants2.template.java -> ../src/descent/core/formatter/DefaultCodeFormatterConstants2.java
# DefaultCodeFormatterOptions2.template.java -> ../src/descent/internal/formatter/DefaultCodeFormatterOptions2.java
# FormatterMessages.template.java -> ../../descent.ui/src/descent/internal/ui/preferences/formatter/FormatterMessages.java
# FormatterMessages.template.properties -> ../../descent.ui/src/descent/internal/ui/preferences/formatter/FormatterMessages.properties
# WhiteSpaceOptions.template.java -> ../../descent.ui/src/descent/internal/ui/preferences/formatter/WhiteSpaceOptions.java
#
# The syntax for inlined perl in the template files is fairly strict - to
# evaluate a block of Perl for each option encountered in this file, start a
# block comment with "/* EVAL-FOR-EACH", which must be on the same line.
# Any lines until a line containing "*/" that start with a "* " (note the space)
# will be evaled as perl. This implies that the line with the "*/" on it is not
# evaled. It's a bit hackish, but it works, right?
#
# The syntax of this file is also pretty strict. If a line starts with a #, that
# line is ignored (or if the line is blank), otherwise it is processed. Lines
# without an "=" are treated as the beginnings of definitions, lines with an "="
# are treated as parts of the definition, with name=value pairings. The first
# line of the definition (the one without the "=" is considered the option name,
# accessed with the key "optName". There's also "constName", which is the
# uppercase version of "optName", with "FORMAT_" appended.
#
# In the inlined Perl, DST is the destination file handle, and $_ is a hash
# reference to the settings for that option. So, $$_{'optName') is the option
# name and if the option definition had a "type" property, $$_{'type'} is used
# to access it.
#
# There's also an EVAL-ONCE now, which forces something to be evaluated only
# when it is encountered. For this, if you need to access a particular option,
# use the @options array of hash references (so $$options[0]{'optName'} is the
# option name. Don't forget to "our" any variables you plan to use again.
#
# Hopefully that isn't too hard to understand (you know, I probably spent longer
# typing this up than it would have taken me to fix the damn script).
#
# For options, these are the meanings of the various hash keys used so far:
#     optName       The name of the option used in an instance of DefaultCodeFormatterOptions
#     constName     The name of the constant string for Java map keys used in DefaultCodeFormatterConstants
#     type          The type of the option in DefaultCodeFormatterOptions
#     default       The default value (we may want to have multiple profiles...?)
#     wsDElem       The name of the D element a white space option refers to (i.e. for_loops, etc.)
#     wsSynElem     The name of the syntax element a white space options refers to (i.e. before_semicolon)
#     wsDElemEx     The D element name displayed in the syntax element tree (defaults to wsDElem)
#     wsSynElemEx   The Syntax element name displayed in the D element tree (defaults to wsSynElem)
#     wsPreview     The preview used for the syntax element
#     javaDefault   Defaults similar to the Java conventions
#     csDefault     Defaults similar to the C# Conventions
#
# The "default" is the "Descent Defaults" option and is also propagated to the
# other defaults if no explicit alternate is given. This means that alternatives
# only need to be defined if they're actually different than the Descent
# default. Java Defaults should conform to the Java Style Guide located at:
# http://java.sun.com/docs/codeconv/html/CodeConvTOC.doc.html
# ... as much as possible (so constructs that appear in Java should be identical
# and D-only constructs should use what would logically follow or the Descent
# defaults). C# defaults are defined similarly, using the conventions here:
# http://blogs.msdn.com/brada/articles/361363.aspx
# Note that that is an incomplete guide for C#, so only more so for D, and thus
# should be adapted as closely as possible.
#
# wsSynElem and wsDElem are meant to be used in tandem - for example, if an
# option is insert_space_before_semicolon_in_for_loops, wsDElem = "for_loops"
# and wsSynElem = "before_semicolon". The names used here refer to the messages
# in FormatterMessages.java. wsPreview is the name of the preview snippet in
# WhiteSpaceOptions.java. wsSynElemEx is the name of the syntax element
# displayed on the by-D-element tree, and is not required. Similarly, wsDElemeEx
# is the name displayed on the by-syntax-element tree and is also not required.
# Usually, if one of wsDElem or wsSynElem is used, the other is needed as well.
# These default to wsSynElem and wsDElem, respectively, if not specified.
#
# Note: there may be more options that those really "needed". For those
# ones, instead of deleting the preference, just set a default and
# don't show them in the UI. If later in time users request that
# option, show them in the UI. :-)
